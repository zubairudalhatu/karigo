# Paystack Test Mode Checkout Verification - Task 101

Date prepared: 2026-07-13

## Purpose

Prepare and verify KariGO checkout in controlled Paystack Test Mode while keeping live
payments disabled.

This pack covers:

- Customer App checkout and pending-payment verification.
- Backend payment initialization, verification and webhook readiness.
- Admin order/payment visibility.
- Vendor payment-status visibility.
- Wallet/refund-readiness notes.
- QA evidence and go/no-go recording.

This task does not activate live Paystack, live payouts, wallet withdrawals, live rides,
Pharmacy marketplace, provider login, or production payment processing.

## Provider Integration Context

| Provider | Intended responsibility | Task 101 status |
| --- | --- | --- |
| Paystack | Customer checkout and payment collection | Test Mode readiness only; no live payment activation |
| Accelerate.ng | Future utility payment service provider for airtime, data, electricity, cable TV and other supported bills/utilities | Documentation context only; no live utility fulfilment |
| Termii | Future SMS provider for OTP, order status SMS, Delivery Captain notifications, utility alerts, wallet/refund alerts and referral notifications | Documentation context only; no live SMS sending |

Paystack Test Mode, Accelerate utility services, and Termii SMS are integration-ready
concepts only. Live payment collection, live utility fulfilment, wallet refund automation,
SMS sending and payout automation remain disabled until separately approved.

## Safety Rules

- Use Paystack Test Mode only.
- Do not commit `.env` files, keys, tokens, webhook secrets, screenshots containing
  secrets, test instruments or payment-provider dashboard credentials.
- Do not paste test card numbers or full provider payloads into Git.
- Do not enable live payment collection.
- Do not activate Accelerate.ng utility transactions or Termii SMS sending in this task.
- Do not mark a payment successful from the frontend. KariGO backend verification is
  the only source of truth.
- Keep mock payment as the rollback provider.

## Required Staging Environment Variables

Store values only in Render/staging secret manager.

```dotenv
PAYSTACK_MODE=test
PAYMENTS_PROVIDER=paystack
PAYMENT_PROVIDER=paystack
PAYMENTS_LIVE_ENABLED=false
PAYSTACK_SECRET_KEY=
PAYSTACK_PUBLIC_KEY=
PAYSTACK_WEBHOOK_SECRET=
PAYSTACK_BASE_URL=
PAYSTACK_CALLBACK_URL=
```

Notes:

- `PAYMENTS_PROVIDER` is the preferred current selector.
- `PAYMENT_PROVIDER` remains supported for backward compatibility.
- `PAYSTACK_SECRET_KEY` must be a test-mode server-side key.
- `PAYSTACK_PUBLIC_KEY` is client-safe only if a future approved client-side flow needs it;
  the current Customer App flow uses only backend-returned authorization data.
- `PAYSTACK_WEBHOOK_SECRET` is server-side only.
- `PAYMENTS_LIVE_ENABLED` must remain `false`.

## Customer App Test Script

### T101-CUS-001 - Mock Regression

1. Keep `PAYMENTS_PROVIDER=mock` and `PAYMENT_PROVIDER=mock`.
2. Customer creates an order.
3. Tap `Continue to payment`.
4. Confirm mock payment verifies immediately.
5. Confirm order detail shows paid status.

Expected result:

- Mock remains usable for demo fallback.
- No Paystack page opens.
- No live provider is called.

### T101-CUS-002 - Paystack Test Authorization

1. Configure backend staging with approved Paystack Test Mode variables.
2. Customer creates an order.
3. Tap `Continue to payment`.
4. Confirm Paystack Test Mode authorization opens externally.
5. Complete provider-approved test checkout outside the repository.
6. Return to KariGO.
7. Tap `Verify payment`.
8. Confirm backend verification changes payment/order status.

Expected result:

- Customer App never sees or stores Paystack secret keys.
- Customer App does not trust callback/query-string status.
- Order is marked paid only after backend verification.

### T101-CUS-003 - Cancelled Or Abandoned Checkout

1. Initiate Paystack Test Mode checkout.
2. Abandon or cancel before successful payment.
3. Return to KariGO.
4. Tap `Verify payment`.

Expected result:

- Verification fails safely or remains pending.
- Order does not move to paid.
- Customer can retry checkout.

## Rollback

1. Set `PAYMENTS_PROVIDER=mock`.
2. Set `PAYMENT_PROVIDER=mock`.
3. Keep `PAYMENTS_LIVE_ENABLED=false`.
4. Disable/remove Paystack staging secrets if appropriate.
5. Redeploy/restart backend.
6. Run mock payment smoke test.
7. Record rollback decision and timestamp.

## Go/No-Go Decision

### Go For Controlled Paystack Test Mode QA

- Paystack Test Mode variables are configured only in staging secret manager.
- Backend rejects missing/non-test/live-enabled Paystack configuration.
- Customer App opens external authorization URL and verifies only through backend.
- Mock fallback still works.
- Admin and Vendor surfaces show safe payment/order status.
- No P0/P1 issues remain.

### No-Go / Hold

- Any live Paystack key or live payment path is used.
- Any secret/test instrument is committed or shown in screenshots.
- Frontend can mark orders paid without backend verification.
- Amount mismatch is accepted.
- Customer cannot complete or retry controlled test checkout.
- Admin/Vendor UI exposes raw provider secrets or sensitive payloads.
